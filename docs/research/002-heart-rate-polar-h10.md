# 002 — Heart-rate: интеграция нагрудного датчика (Polar H10)

**Статус:** research (не реализовано)
**Дата:** 2026-07-08
**Вопрос:** можно ли автоматически (zero-touch, как с дорожкой) подхватывать
нагрудный HR-датчик **Polar H10** по BLE, писать непрерывные сэмплы пульса
рядом с шагами и вывести пульс в T-Max/tmux-виджет?

**Короткий ответ:** да, все три части реализуемы, риск низкий. Polar H10 отдаёт
пульс через **стандартный** GATT-профиль **Heart Rate Service (HRS)**, т.е.
ровно тот же паттерн, что и FTMS-дорожка, — свой сервис в рекламе, notify без
pairing/bonding. Дорабатываем существующий одно-central демон до владельца
**двух** параллельных линков.

---

## 1. Профиль устройства Polar H10

- **Heart Rate Service `0x180D`** — стандартный BLE-профиль. Polar H10 **рекламирует
  этот service-UUID**, значит фильтруется в скане так же, как дорожка по `0x1826`.
- **Heart Rate Measurement `0x2A37`** (notify) — основной поток. Формат кадра
  идентичен по духу Treadmill Data: 1-й байт — флаги, дальше поля по битам.
- **Body Sensor Location `0x2A38`** (read, опционально) — «chest».
- **Открытое подключение.** HRS не требует pairing/bonding — это ключ к zero-touch:
  подключились к GATT и подписались, как к дорожке. (У Polar есть ещё проприетарный
  **PMD**-сервис для ECG/ACC/сырых RR — он нам не нужен; RR-интервалы при желании
  приходят и через стандартный `0x2A37`, бит 4 флагов.)
- **Питание/реклама:** H10 включается от контакта с кожей (электроды должны быть
  влажными) и рекламируется только когда надет; снят → реклама пропадает. Это
  естественный аналог «дорожка off» — демон просто не находит peripheral.
- **До 2 одновременных BLE-подключений** на H10 (плюс 5 кГц GymLink-аналог). Наш
  демон займёт один слот; телефон Polar Flow — второй, если параллельно.

### Формат `0x2A37` (Heart Rate Measurement)

```
byte 0: flags
  bit0  — формат значения: 0 = bpm в u8, 1 = bpm в u16 (LE)
  bit1-2— sensor contact (supported / detected)
  bit3  — energy expended present (u16 kJ)
  bit4  — RR-intervals present (по одному u16 каждый, единица 1/1024 s)
byte 1..: bpm (u8 или u16 по bit0), далее опц. energy, далее RR[]
```

H10 обычно шлёт bpm как u8 + флаг sensor-contact + (опц.) RR. Парсер — калька с
`parse_treadmill_data` (`src/ftms.rs:114`): читаем флаги, курсор, условные поля
через те же `read_u8/u16` хелперы (`src/ftms.rs:200`).

## 2. Zero-touch авто-подключение

**Вывод: полностью совпадает с моделью дорожки, отдельного «спаривания» не нужно.**

| Шаг | Дорожка (сейчас) | HR (по образцу) |
|---|---|---|
| Скан по service-UUID | `ScanFilter{services:[0x1826]}` `scan.rs:142` | `[0x180D]` |
| Матч peripheral | `is_treadmill()` `scan.rs:186` | `services.contains(0x180D)` |
| Connect + discover | `scan.rs:150-168` | тот же код |
| Subscribe notify | `0x2ACD` `scan.rs:199` | `0x2A37` |

**Два линка с одного central — поддерживается.** Ограничение «один central на
peripheral» (`daemon.rs:11`) относится к стороне *дорожки* (её нельзя держать и
демоном, и телефоном), а не к числу периферий на нашем central. btleplug/
CoreBluetooth держат несколько подключённых `Peripheral`, у каждого свой
`.notifications()`-стрим.

**Нюанс скана:** CoreBluetooth фильтрует по service-UUID из рекламы, а `ScanFilter`
сейчас жёстко под `0x1826`. Варианты: (а) отдельный скан-проход/фильтр под HR, либо
(б) сканировать без фильтра и матчить оба типа. Проще — отдельная `connect_hr()`
рядом с `connect_treadmill()` (`scan.rs:140`), запускаемая параллельно.

**Идентификация конкретного датчика:** на macOS адреса нет (непрозрачный system
UUID). Первый найденный HR-peripheral нас устраивает (как с дорожкой). Если рядом
несколько — позже добавить фильтр по `local_name` (H10 рекламируется как
`Polar H10 XXXXXXXX`); пока YAGNI.

## 3. Хранение: непрерывные сэмплы пульса рядом с шагами

Новая таблица по образцу `raw_samples` (`store.rs:206`), привязка к той же
treadmill-`session_id` (пульс имеет смысл во время работы дорожки; если датчик
надет без дорожки — либо отдельная HR-сессия, либо `session_id NULL`):

```sql
CREATE TABLE IF NOT EXISTS hr_samples (
  id         INTEGER PRIMARY KEY,
  session_id INTEGER REFERENCES sessions(id),
  ts_ms      INTEGER NOT NULL,
  bpm        INTEGER NOT NULL,
  rr_ms      BLOB,            -- опц. RR-интервалы, для HRV; можно отложить
  raw_frame  BLOB NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_hr_samples_session ON hr_samples(session_id);
```

- Вставить в `migrate()`-батч около `store.rs:217`; инсертер `insert_hr_sample()`
  по образцу `insert_raw_sample` (`store.rs:480`).
- Запись — в новой notify-ветке демона (см. §4), рядом с `store.insert_raw_sample`
  (`daemon.rs:513`). Кадры прилетают ~1 Гц → нагрузка на БД пренебрежима.
- **Джойн с шагами** идёт по `session_id` + `ts_ms` (тот же масштаб, что у
  `raw_samples.ts_ms`), так что HR и шаги ложатся на одну временную ось. Это
  открывает будущие фичи: средний/макс пульс за тренировку, HR-зоны, HRV.

## 4. Демон: второй стрим в цикле

`stream_with_presence` (`daemon.rs:383`) владеет `peripheral` дорожки до
дисконнекта. Добавляем:

1. После подписки на дорожку (`daemon.rs:393`) — попытка `connect_hr()` +
   `subscribe_hr()` (**best-effort**: нет датчика → просто без пульса, WARN, не
   роняем основной цикл — датчик надевают не всегда).
2. Второй `.notifications()`-стрим HR-peripheral, новая ветка в `tokio::select!`
   около `daemon.rs:485`: парс `0x2A37` → `store.insert_hr_sample(...)` →
   обновить `last_bpm` в `DaemonState` (`daemon.rs:995`).
3. Последний bpm + его свежесть класть в снапшот `daemon_status`, чтобы `widget`
   читал пульс **без BLE** (как всю остальную статистику).
4. **Lifecycle:** HR-линк закрывать в тех же exit-путях, что и treadmill-линк
   (`disconnect_best_effort` `daemon.rs:364`) — не оставлять висящий peripheral;
   отдельный watchdog/timeout на HR-стрим по образцу `NOTIFICATION_TIMEOUT`
   (`daemon.rs:485`), чтобы пропавший датчик не держал ветку.

Замечание: FTMS Treadmill Data уже имеет бит `HEART_RATE_PRESENT` (`ftms.rs:99`),
но дорожка Yesoul пульс не отдаёт (поле пропускается, `ftms.rs:172`) — поэтому
именно отдельный нагрудный датчик, а не поле дорожки.

`run_widget` печатает TSV из 8 полей, только чтение из SQLite (`main.rs:628`).
Добавляем поле `bpm` (9-е), источник — `last_bpm` из `daemon_status`:

- В TSV — ещё одно числовое поле; пусто/`-` если пульс stale или датчик не надет

- Обновить контракт «8 полей» в докстринге (`main.rs:563`) и потребителя виджета

## 6. Оценка объёма и риски

**Объём:** ~1 задача уровня 014/015. Новые: HR-константы (`ftms.rs`/новый `hr.rs`),
`connect_hr`/`subscribe_hr` (`scan.rs`), `parse_hr_measurement`, таблица+инсертер
(`store.rs`), ветка в демоне, поле в widget. Существующие паттерны переиспользуются
почти целиком.

**Риски / что проверить на железе:**
- Реально ли H10 рекламирует `0x180D` в скане на macOS/CoreBluetooth (ожидаемо да).
- Порядок: подключать HR до/после/параллельно дорожке — не мешают ли два `connect`
  под одним `start_scan`; возможно нужен раздельный скан-проход.
- Поведение при снятии датчика в середине тренировки (реклама пропадает →
  reconnect-петля, не должна ломать основной цикл дорожки).
- Занятость второго BLE-слота H10 телефоном Polar Flow (тогда наш connect может не
  пройти — сообщить WARN, деградировать без пульса).

**Что НЕ делаем сейчас (YAGNI):** PMD/ECG/сырой ACC, HRV-аналитика, HR-зоны,
идентификация конкретного датчика по имени, отдельная HR-сессия без дорожки.
Задел под RR-интервалы (колонка `rr_ms`) закладываем, но парсинг можно отложить.
