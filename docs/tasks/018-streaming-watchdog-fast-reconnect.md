# 018 — Быстрый watchdog при обрыве во время стриминга

## Инцидент (2026-07-06, live)

Оператор выдернул дорожку из розетки на 10–30 с и быстро вернул. Демон ~2 минуты
не трекал ходьбу и не выставил скорость, пока не восстановился.

Разбор логов (UTC; локаль = +8ч):
- **17:31:19** — обрыв питания. Телеметрия оборвалась.
- `NOTIFICATION_TIMEOUT` (20 с) **не сработал** — не было ни `warn` о
  «no telemetry», ни реконнекта. Значит executor-поток был **заблокирован внутри
  blocking-FFI CoreBluetooth/btleplug** (быстрый power-cycle подвешивает вызов),
  и tokio-таймер на этом же потоке не мог сработать (тот самый «тихий висяк» из
  задачи 007).
- **17:33:32** — watchdog (на отдельном потоке) поймал `stale_s=133 > 120` →
  `exit(86)` → launchd рестартнул.
- **17:33:36** — реконнект, трекинг возобновился (self-heal сработал).

**Корень:** watchdog-порог `WATCHDOG_STALE_THRESHOLD = 120 с` консервативен, т.к.
покрывает и фазу scan/connect (легитимные паузы до ~40 с: 15 scan + 20 connect +
5 retry). Но во время **активного стриминга** дорожка шлёт ~1/с, поэтому молчание
даже >40 с при живой ссылке невозможно — это гарантированно мёртвый линк.

## Фикс — порог, зависящий от фазы

Watchdog получает флаг `streaming` (`AtomicBool`, как `last_touch_ms`). Порог
выбирается по фазе:
- **стриминг активен** → `STREAMING_STALE_THRESHOLD = 40 с` (> `NOTIFICATION_TIMEOUT`
  20 с, чтобы штатный путь успел сработать первым, когда executor **не** заблокирован;
  ≪ 120 с);
- **иначе** (scan/connect/subscribe/teardown) → прежний `WATCHDOG_STALE_THRESHOLD = 120 с`.

Проводка:
- `set_streaming(true)` + `touch()` в `stream_with_presence` **после** открытия
  notification-стрима (фаза subscribe/connect остаётся под 120-с порогом — там
  bounded-таймауты уже есть, а хэнги реже).
- `set_streaming(false)` в `run()` **сразу после** возврата из
  `stream_with_presence` (покрывает все выходы: `?`-ошибки, break, ранний
  `return` на WillPowerOff), до потенциально-долгого `disconnect_best_effort`
  (его хэнг снова под 120-с — как в задаче 007).

Эффект: окно «ходьба без трекинга» при обрыве во время ходьбы падает с ~133 с до
~44 с (40 с порог + ~4 с реконнект). Ложные срабатывания практически исключены
(1/с стрим). Ложный exit дёшев (~4 с реконнект; сегмент склеится read-time
`merge_segments`). `Instant` на macOS не тикает во сне — засыпание mid-stream не
даёт false-positive (как и раньше).

## Что НЕ меняем

- Blocking-FFI внутри btleplug не чиним — CoreBluetooth-хэнг неисправим in-process
  (задача 007). Рестарт через launchd — принятый паттерн.
- `NOTIFICATION_TIMEOUT` (20 с) остаётся штатным быстрым путём для случая, когда
  executor **не** заблокирован (чистый обрыв): он даёт `treadmill_lost` toast +
  аккуратный реконнект без рестарта процесса.

## Затронутые файлы

- `src/daemon.rs` — `STREAMING_STALE_THRESHOLD`, поле `streaming` в `Watchdog`
  + `set_streaming`/`stale_threshold`, порог в `is_stale_at`/`spawn_monitor`,
  проводка в `run()`/`stream_with_presence`. Тест на пороги.
