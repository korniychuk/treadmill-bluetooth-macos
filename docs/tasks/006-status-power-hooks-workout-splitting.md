- `pmset -g batt` в этот момент стабильно показывал `AC Power` — то есть
  зависший процесс либо застрял в ветке "на батарее, скан пропускаем"
  (`daemon.rs::run`, добавлена в 005) вопреки реальному питанию, либо
  poll-цикл (`BATTERY_POLL_INTERVAL` = 60s, опрос через дочерний процесс
  `pmset`) не восстановился штатно после смены источника питания.
- Что усугубило диагностику: причина простоя логируется на уровне `debug!`

  его не показывает — снаружи процесс выглядел просто "молчащим", без единой
  подсказки, что происходит и почему.
- **Временный фикс:** `launchctl kickstart -k` — демон переподключился сразу
  же после рестарта.

Корневая причина — вся энергосберегающая логика построена на **поллинге**
(`pmset` раз в 60 секунд) вместо подписки на реальные системные события. Это
и хуже с задержкой реакции, и — судя по живому инциденту — оставляет шанс
залипания в устаревшем состоянии без какого-либо диагностируемого сигнала
снаружи. Отсюда все три задачи ниже.

## Задача A — event-driven power/sleep хуки вместо поллинга `pmset`

Заменить `power::is_on_ac_power()` (poll раз в 60с через `Command::new("pmset")`)
на подписку на нативные macOS-уведомления через IOKit, чтобы демон реагировал
на изменение питания/сна **мгновенно**, а не с задержкой до 60 секунд, и чтобы
не было класса багов "поллинг где-то застрял и не заметил переход".

Полный список хуков, которые нужно собрать и обработать:

1. **Power source changed** (кабель воткнут/выдернут, AC ⇄ battery) —
   `IOPSNotificationCreateRunLoopSource` (IOKit `IOPowerSources.h`). Заменяет
   текущий `pmset`-поллинг.
2. **System will sleep** — `IORegisterForSystemPower` +
   `kIOMessageSystemWillSleep`. Нужно **подтвердить** сон через
   `IOAllowPowerChange` (иначе система будет ждать таймаут/блокировать sleep).
   На этом событии: если сессия активна — не рвать соединение самим (BLE
   всё равно ляжет физически), но залогировать переход на INFO и обновить
   персистентный статус (задача B), чтобы `status` мог честно показать
   "ушёл в сон в HH:MM".
3. **System has powered on / did wake** — `kIOMessageSystemHasPoweredOn`.
   На этом событии — форсировать немедленную проверку питания и попытку
   скана (не ждать никакого таймера), залогировать на INFO.
4. **System will power off / restart** — `kIOMessageSystemWillPowerOff` /
   `kIOMessageSystemWillRestart` — best-effort: закрыть текущую сессию в БД
   (`store.end_session()`), чтобы не оставлять "зависшую" незакрытую сессию.

### Архитектура

IOKit-колбэки живут на `CFRunLoop` — они не async/tokio-совместимы напрямую.
Стандартный паттерн: отдельный OS-поток, на нём создаётся run loop, вешаются
источники (`IOPSNotificationCreateRunLoopSource`,
`IONotificationPortGetRunLoopSource` для power-management), `CFRunLoopRun()`
блокирует поток навсегда. Колбэки шлют события в tokio-мир через
`tokio::sync::mpsc` (или `std::sync::mpsc` + `Handle::spawn`), которые
`daemon::run()` читает вместо текущего `sleep(BATTERY_POLL_INTERVAL)`.

Предлагаемый новый модуль `src/power.rs` (переписать, не патчить точечно):

```rust
pub enum PowerEvent {
    AcPowerChanged(bool),   // true = on AC
    WillSleep,
    DidWake,
    WillPowerOff,
}

pub fn spawn_power_event_listener() -> tokio::sync::mpsc::UnboundedReceiver<PowerEvent> { ... }
```

Реализация — unsafe FFI к `IOKit.framework`/`CoreFoundation.framework`
(`io-kit-sys` + `core-foundation`/`core-foundation-sys` крейты, либо ручные
`extern "C"` биндинги — на усмотрение исполнителя, проверить, что реально
собирается и линкуется на macOS/aarch64 из `Cargo.toml`). `build.rs` должен
задать `cargo:rustc-link-lib=framework=IOKit` и `=CoreFoundation`.

**Важно:** каждый переход обязательно логируется на уровне `info!`, не
`debug!` — цель как раз в том, чтобы больше никогда не было "тихого" перехода
в необъяснимое состояние (см. инцидент выше). Частые/повторяющиеся события не
нужны — только переходы состояния, это не тикающий поллинг.

`daemon::run()` переписывается на `tokio::select!` между: событием от
power-listener, результатом `scan::connect_treadmill`, и текущей веткой
работы с найденной дорожкой — вместо текущего `if !is_on_ac_power() { sleep }`.

Тесты: `power.rs` теперь FFI-обёртка без чистой логики для юнит-тестов —
де-риск на реальном железе обязателен (voltage plug/unplug, закрыть крышку)
перед тем, как считать задачу готовой, аналогично де-риску в 005.

## Задача B — CLI-команда `status`

Новая подкоманда, читает персистентное состояние (без попытки самой открыть
BLE-адаптер — `status` не должен конкурировать с демоном за адаптер).

Демон (`daemon.rs`) должен на каждый переход писать текущий статус в новую
таблицу SQLite `daemon_status` (id=0, всегда один ряд, upsert — по образцу
`device_baseline`):

```sql
CREATE TABLE daemon_status (
  id INTEGER PRIMARY KEY CHECK (id = 0),
  connected INTEGER NOT NULL,       -- 0/1 — есть сейчас BLE-соединение
  presence_state TEXT,              -- Walking | Paused | AwayWhileRunning | Unknown | NULL
  last_connected_at TEXT,           -- RFC3339, последний раз когда стало connected
  last_disconnected_at TEXT,        -- RFC3339, последний раз когда стало disconnected
  power_mode TEXT NOT NULL,         -- 'ac_scanning' | 'battery_idle'
  power_mode_since TEXT NOT NULL,   -- RFC3339, с какого момента в этом режиме
  updated_at TEXT NOT NULL
);
```

Вывод `status` должен максимально подробно покрывать:

1. **Состояние дорожки:** подключена ли сейчас, если да — presence
   (Walking/Paused/AwayWhileRunning), с какого момента; если нет — когда была
   подключена в последний раз (`last_disconnected_at`, human-readable "N
   минут/часов назад").
2. **Демон:** жив ли процесс вообще (проверить через `launchctl print
   gui/$UID/com.korniychuk.treadmill-bluetooth-macos.daemon` или аналог, или
   PID-файл — на усмотрение исполнителя; не полагаться только на записи в БД,
   которые могут быть устаревшими от мёртвого процесса).
3. **Режим сканирования:** сейчас активно сканируем (на AC) или в
   энергосберегающем режиме (на батарее); с какого момента в этом режиме.
   После задачи A это событийное состояние, а не "раз в N секунд" — так и
   написать (например: "battery idle since HH:MM — will resume immediately on
   AC power or wake" вместо несуществующего интервала).
4. **Условие выхода из энергосберегающего режима:** появление AC-питания
   (событие из задачи A) — явно перечислить триггер в выводе, а не только
   молчаливо в коде.
5. **Тренировки за сегодня** (см. задачу C) — список с суммами по каждой, и
   если прямо сейчас идёт тренировка — пометить её как текущую/незавершённую.
6. **Здоровье демона (см. задачу D):** если `daemon_status.updated_at` старше
   watchdog-порога, а процесс при этом жив — явно показать это как
   подозрение на hang, а не молча печатать устаревшие данные как актуальные.

Формат вывода — человекочитаемый текст (как у существующего `stats`), не
JSON — если оператору позже понадобится машиночитаемый вывод, это отдельная
задача (`--json`), сейчас не запрашивалась.

## Задача C — разделение тренировок внутри дня (порог 15 минут)

Сейчас `daily_stats` — один агрегат на календарный день. Нужно: если разрыв
между реальной ходьбой (presence = Walking) превышает **15 минут**, это уже
отдельная тренировка; каждая тренирока считается и хранится отдельно, дневная
сумма остаётся производной (суммой тренировок за день).

Новая таблица:

```sql
CREATE TABLE workouts (
  id INTEGER PRIMARY KEY,
  date TEXT NOT NULL,             -- YYYY-MM-DD локальная дата начала
  started_at TEXT NOT NULL,       -- RFC3339
  ended_at TEXT NOT NULL,         -- RFC3339, обновляется на каждый кредит
  distance_m INTEGER NOT NULL DEFAULT 0,
  steps INTEGER NOT NULL DEFAULT 0,
  walking_time_s INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_workouts_date ON workouts(date);
```

Логика (в `store.rs`, рядом с `credit_daily` — само решение "разделять или
нет" на основе времени последней активности, не привязано к BLE-сессии:
оператор может отключиться/переподключиться в рамках одной тренировки, а
может уйти на реальный перерыв внутри одной BLE-сессии):

- Персистентно хранить `last_activity_at` (последний момент реального
  кредита Walking — не привязан к сессии/коннекту).
- На очередном кредите: если `now - last_activity_at > 15 минут` (или нет
  активной тренировки вообще) — закрыть предыдущую (если была) и открыть
  новую строку `workouts`; иначе — обновить существующую (агрегаты + `ended_at`).
- `daily_stats` не убирается (обратная совместимость `stats` без флагов), но
  становится обновляемой тем же кредитом — оба апдейта в одной транзакции.

`stats` (и `status` для сегодня) должны показывать разбивку по тренировкам, не
только дневную сумму — включая исторические дни (`stats --all` / `stats --date
YYYY-MM-DD`, на усмотрение исполнителя, добавить флаг если понадобится для
конкретного дня).

Тесты: аналогично существующим тестам `credit_or_hold` — зафиксировать оба
сценария (продолжение тренировки vs. разрыв > 15 мин ⇒ новая запись), плюс
кейс "первая тренировка дня" (нет предыдущей активности).

**Edge case — переход через полночь.** Если тренировка идёт через 00:00,
кредиты по факту уходят в `daily_stats` за два разных календарных дня, а
`workouts.date` — одно значение. Решение: `workouts.date` фиксируется по
`started_at` (дата начала тренировки) и не меняется до её закрытия — то есть
"дневная сумма = сумма тренировок дня" **не** гарантируется инвариантом для
дня, в который тренировка закончилась, только для дня, в который она
началась. Зафиксировать явно тестом (тренировка 23:50→00:10 ⇒ один
`workouts`-ряд с `date` = день начала, `daily_stats` — два дня с разными
частями).

## Задача D — watchdog на silent hang + таймаут на `connect`/`discover_services`

Диагностика инцидента не смогла на 100% объяснить расхождение (CPU будто
поллит, `pmset` руками показывает AC, но скана не было 3+ минуты) — то есть
остаётся класс зависаний **внутри `btleplug`/CoreBluetooth**, который задача A
(убирает только `pmset`-поллинг) не закрывает. Оператор явно просил "чтобы не
было такого, что сервис уснул и не проснулся вовремя" — это требование
покрывает такие hang'и тоже, не только power-gate.

1. **Таймаут на подключение.** В `scan.rs::connect_treadmill` /
   `connect_by_id` вызовы `peripheral.connect().await` и
   `peripheral.discover_services().await` сейчас ничем не ограничены по
   времени — `SCAN_TIMEOUT` бьёт только по циклу поиска, не по самому
   подключению. Обернуть оба вызова в `tokio::time::timeout` (отдельная
   константа, например `CONNECT_TIMEOUT`), чтобы зависший CoreBluetooth-вызов
   не вешал весь демон навсегда.
2. **Watchdog на `daemon_status.updated_at`.** Демон обновляет
   `daemon_status.updated_at` на каждом цикле (найден/не найден/сэмпл/переход
   питания — что угодно, лишь бы регулярно). Отдельная лёгкая задача внутри
   `daemon::run()` (или дочерний tokio-таск) логирует `WARN` на уровне info,
   если с последнего обновления прошло дольше разумного порога (например,
   `SCAN_TIMEOUT + CONNECT_TIMEOUT` с запасом) — это ловит **любой** тихий
   hang независимо от причины, а не только power-gate. Не пытаемся
   автоматически "лечить" зависание (это отдельный риск сам себя убить не по
   делу) — только видимый сигнал, что что-то не так, для `status` и логов.
