# 033 — Снятый Polar H10 «замораживает» bpm: сердце в виджете и мусор в статистике

**Статус:** план
**Тип:** баг, приоритет высокий (портит уже записанные данные)

## Симптом

Нагрудный датчик снят и лежит на столе. В tmux-виджете по-прежнему горит
`♥ 111`, `tm status` печатает `heart rate: sensor connected, 111 bpm`, а
`hr_samples` продолжает наполняться значением 111 раз в секунду. Итог:
тренировка в `tm status` показывает `♥ 111/111` — avg и max сошлись, потому
что половина окна это одна и та же замороженная константа.

## Доказательство

Сырые кадры `0x2A37` из `hr_samples.raw_frame`:

```
надет:  10 75 72 03 27 02    flags=0x10 (RR present), bpm=0x75=117, RR=...
надет:  10 75 0B 01 9A 01    flags=0x10,              bpm=117
снят:   00 6F                flags=0x00,              bpm=0x6F=111
снят:   00 6F                ← ежесекундно, бит-в-бит одинаковый кадр
```

## Причина

Три независимых предположения, каждое из которых неверно для H10:

1. **`hr.rs` считает, что потеря контакта = `bpm == 0`.** H10 в этом режиме
   шлёт последнее валидное значение, а не ноль. `parse_hr_measurement` его
   пропускает как нормальный кадр.
2. **`contact: Option<bool>` парсится, но нигде не используется** — и всё равно
   не помог бы: в кадре `00 6F` бит `CONTACT_SUPPORTED` сброшен, значит
   `contact == None`, «датчик не умеет репортить контакт».
3. **`HR_NOTIFICATION_TIMEOUT` (10 с) — сторож молчания, а не сторож смысла.**
   Кадры идут ежесекундно, таймаут не срабатывает никогда; `state.last_bpm_ts`
   обновляется, `hr_connected` остаётся `true`.

Плюс отдельная слабость виджета: `widget_hr_field` считает bpm свежим, пока
ему меньше `WATCHDOG_STALE_THRESHOLD_S` (= 15+20+60 = **95 с**). Порог
рассчитан на «демон повис» (scan + connect + запас), для пульса он абсурдно
велик: даже при корректной детекции сердце висело бы полторы минуты.

## Настоящий сигнал

При потере контакта H10 перестаёт слать **RR-интервалы**: `flags` схлопывается
`0x10` → `0x00`. Это наблюдаемо, детерминированно и приходит в первом же
«мёртвом» кадре — на порядок раньше любой эвристики про константность bpm.

## Решение

Ввести понятие **контакта**, отдельное от **линка**. Линк (BLE) жив — датчик
подключён; контакт мёртв — телу он не принадлежит. Виджет/статистика смотрят
на контакт, реконнект-логика — на линк.

### `hr.rs` (чистый слой, юнит-тестируемый)

- Оставить `bpm == 0` как есть (валидный сигнал у других страпов).
- Не хардкодить «H10 = RR». Ввести per-session capability: если датчик хоть раз
  прислал кадр с RR (`flags & RR_PRESENT`), значит он их поддерживает.
- Чистая функция-детектор состояния контакта, без времени и BLE внутри:

  ```rust
  pub struct ContactTracker { rr_capable: bool, frames_without_rr: u32 }
  pub enum Contact { Live, Lost }
  impl ContactTracker { pub fn observe(&mut self, m: &HrMeasurement) -> Contact }
  ```

  Правила `observe`, по убыванию надёжности:
  1. `m.contact == Some(false)` → `Lost` немедленно (страпы, которые честно
     репортят контакт).
  2. `rr_capable && m.rr_ms.is_empty()` → счётчик; `frames_without_rr >=
     HR_CONTACT_LOST_FRAMES` (**3**, т.е. ~3 с) → `Lost`. Один-два кадра без RR
     допускаем: сплиты/пропуски бывают.
  3. Иначе `Live`; любой кадр с RR обнуляет счётчик и взводит `rr_capable`.

  Датчик без RR и без contact-битов детектировать нечем — он остаётся `Live`
  до `HR_NOTIFICATION_TIMEOUT`. Это честная деградация, а не регресс: сегодня
  так ведут себя все.

### `daemon.rs`

- HR-ветка: прогнать кадр через `ContactTracker`. При `Lost`:
  - **не писать сэмпл** в `hr_samples` (иначе агрегаты снова отравятся),
  - `state.hr_connected = false`, `last_bpm`/`last_bpm_ts` → `None`,
  - `warn!` один раз на переход `Live → Lost` (не каждую секунду), с bpm и
    числом кадров без RR,
  - **линк не рвать**: `hr_notifications`/`hr_peripheral` остаются, батарея
    остаётся. Оператор вернул датчик на грудь → RR вернулись → `Live`, без
    15-секундного рескана. `hr_connected = true`, `info!` о возврате.
  - `ContactTracker` сбрасывается вместе с остальным HR-состоянием при реальной
    потере линка (`Ok(None)` / таймаут).

### `main.rs` (виджет / status)

- Отдельная константа `HR_STALE_THRESHOLD_S: i64 = 15` вместо
  `WATCHDOG_STALE_THRESHOLD_S` в `widget_hr_field`. Кадры идут ~1/с, линк-таймаут
  10 с — 15 с даёт запас ровно на один пропущенный цикл реконнекта и ни секундой
  больше. `widget_speed_field` (задача 029) переиспользует **свой** порог, его
  не трогаем: телеметрия ленты имеет другую частоту.

Дублировать детекцию в `main.rs` (например, «bpm не менялся N секунд») не надо:
единственный писатель `daemon_status` — демон, ему и решать.

## Тесты

- `ContactTracker`: RR → нет RR ×2 → всё ещё `Live`; ×3 → `Lost`; возврат RR →
  `Live`, счётчик обнулён.
- `contact == Some(false)` → `Lost` с первого кадра, даже с RR.
- Датчик, никогда не присылавший RR (`rr_capable == false`), не уходит в `Lost`.
- Реальные кадры из этого бага как фикстуры: `10 75 72 03 27 02` → `Live`,
  затем `00 6F` ×3 → `Lost`.
- `widget_hr_field`: bpm возрастом 20 с → пусто.

## Чистка данных

Записанные фальшивые сэмплы уже искажают `hr_summary_for`. Отдельный вопрос к
оператору — оставляем как есть или чистим точечным `DELETE` по окну.
`recompute-segments` (задача 015) их не трогает: он про `activity_segments`,
не про `hr_samples`.
