# 015 — `tm recompute-segments --from-raw`

## Мотивация

Задача 014 засидила `activity_segments` из **агрегированной** `workouts` (1 старая
тренировка → 1 грубый сегмент): внутренняя структура (паузы/сходы внутри
тренировки) в `workouts` уже схлопнута. Но `raw_samples` хранит **каждый кадр** —
ground truth. Значит сегменты можно пересобрать из него **точно**, проиграв ту же
presence/credit-логику, что и живой демон.

Это не только чинит грубую историю — это **общий инструмент** пересборки
производных данных из raw при любом будущем изменении логики расчёта (напр. если
однажды поменяется away-порог): raw = истина, derived = пересобираемо по кнопке.

## Что делаем

Новая CLI-команда **`tm recompute-segments`** (read-only по BLE, как `stats`):
- читает все `raw_samples` в порядке `ts_ms`;
- проигрывает **ту же** presence + credit логику, что живой демон, но по
  историческим таймстампам;
- перестраивает таблицу `activity_segments` с нуля (транзакционно, идемпотентно);
- НЕ трогает `raw_samples` (истина), `daily_stats` (уже календарно-верна, живая),
  `workouts` (архив 014).

## Ключевой принцип: переиспользовать живую логику, НЕ форкать

Матчить поведение живого демона можно только одним кодом. Значит:

1. **`PresenceTracker` делаем time-injectable.** Сейчас `observe()` дёргает
   `Instant::now()` для away-порога. Рефактор: принимать явное «now» (или
   монотонное время), чтобы:
   - живой путь передавал `Instant::now()` (поведение НЕ меняется — проверить
     существующими тестами presence);
   - replay передавал время, синтезированное из `ts_ms` кадров.
   Это единственный источник правды для 10с-away-логики.
2. **Credit/segment-эмиссия — переиспользуемый «движок».** Логика
   `credit_or_hold` (pending-буфер, discard-on-away, подтверждение шагом) +
   открытие/продление/закрытие сегмента должна вызываться **и** живым циклом,
   **и** replay-командой из общего места, а не дублироваться.

## Тонкости replay из raw

- **Сброс счётчиков устройства.** `raw_samples` хранит СЫРЫЕ кумулятивные
  счётчики (сбрасываются в 0 при power-cycle/новом старте ленты). Дельты считать
  по-кадрово с обработкой сброса (delta = current, если current < prev) — см.
  существующую `store::raw_distance_m` как образец для distance; аналогично для
  шагов/времени.
- **Тайминг.** Presence away-порог = 10с. В replay «время» = `ts_ms` кадров.
  Кадры идут ~1/с; разрывы между сессиями (реконнекты) могут быть большими —
  presence на новой сессии стартует свежим (как в живом коде — новая сессия =
  свежий трекер).
- **Границы сессий.** Разные device-сессии в `raw_samples` (`session_id`).
  Обрабатывать посессийно, presence/pending сбрасывать между сессиями (как живой
  демон при реконнекте).

## Безопасность / UX

- Транзакция: собрать новые сегменты, затем заменить содержимое
  `activity_segments` атомарно. Идемпотентно (повторный запуск = тот же
  результат).
- Вывести сводку: сколько кадров обработано, сколько сегментов получилось, за
  какой период. Лог edge-cases (undecodable/пустые кадры, сбросы) — WARN.
- Не требует BLE-адаптера (как `stats`/`status`).

## Тесты

- Чистый replay-движок: синтетическая последовательность `(ts, speed, steps)` →
  ожидаемые сегменты (ходьба → пауза → ходьба → сход → ходьба; проверить границы,
  discard-on-away, сброс счётчика).
- Регрессия: `PresenceTracker` с инъекцией времени даёт те же переходы, что и
  раньше (существующие presence-тесты зелёные).
- Идемпотентность: два прогона подряд → идентичный `activity_segments`.

## Вне scope

- `daily_stats` не пересобираем (уже верна; риск двойного счёта). Если позже
  захотим — отдельно.
- Полуночный `workout_count`-нюанс из 014 (виджет single-режим на переходящей
  тренировке) — отдельная мелочь, не здесь.
- Авто-запуск recompute (пока только ручная команда «по кнопке»).

## Реализация (что сделано)

### Общий движок (не форк) — `src/activity.rs`
- Извлечён новый модуль `activity.rs` с `PendingCredit`, `credit_or_hold` и
  новым `ActivityAccumulator` (presence + pending + `current_segment`). И живой
  демон (`stream_with_presence`), и replay гоняют **один** `ActivityAccumulator`
  — сегментация идентична by construction.
- `ActivityAccumulator::observe(now, speed, steps)` прогоняет presence и
  **закрывает** открытый сегмент при уходе из `Walking` (перенесено из инлайна
  демона); возвращает transition (демон дёргает по нему toast/speed-restore,
  replay игнорирует). `.credit(store, now, deltas)` вызывает `credit_or_hold`.
- `credit_or_hold` теперь принимает `now: DateTime<Utc>` (устранён TODO в
  demon.rs): живой путь передаёт `Utc::now()` (байт-в-байт как раньше), replay —
  `ts_ms` кадра. Оба пути идентичны.

### PresenceTracker time-injection — `src/presence.rs`
- `observe(now: Instant, speed, steps)` — `Instant::now()` больше не читается
  внутри. Живой демон передаёт `Instant::now()` (поведение не изменилось,
  presence-тесты зелёные после правки сигнатуры). Replay синтезирует `Instant`
  из `ts_ms` (per-session offset от `anchor`, presence сбрасывается посессийно,
  поэтому межсессийная величина не важна и нет `Instant + огромный Duration`).

### Replay — `src/recompute.rs`
- Читает `store.raw_samples_ordered()` (`ORDER BY ts_ms, id`), проигрывает через
  **scratch in-memory `Store`**: `advance_baseline` (переиспользован verbatim —
  та же `delta_since` reset-логика, что и живой) + `credit_activity` (та же
  open/extend/date-логика). Мусорные записи scratch `daily_stats` выбрасываются —
  **реальный `daily_stats` не тронут**. Это ключ: ноль форка, `credit_activity`
  переиспользован целиком.
- **Посессийно:** свежий `ActivityAccumulator` на каждый `session_id` (presence/
  pending/segment сброшены), но scratch `device_baseline` **непрерывен** между
  сессиями — ровно как единственная persisted-строка живого демона (никогда не
  сбрасывается на реконнекте). Сессии не переплетаются (один BLE-central), смена
  `session_id` = чистая граница.
- **Почему совпадает с живым:** `raw_samples.ts_ms` штампуется `Utc::now()` в той
  же итерации цикла демона, что и `Utc::now()` живого кредита, поэтому подача
  `ts_ms` обратно даёт start/end сегмента и local-date-атрибуцию с точностью до мс.
- Транзакционная замена: `store.replace_activity_segments(&segments)` —
  `DELETE` + `INSERT` с явными id в одной транзакции. Replay детерминирован
  (scratch свежий, id 1..N), поэтому **идемпотентно**: два прогона = идентичные
  строки (id + все колонки).
- **Сброс счётчиков** обрабатывается `advance_baseline`/`delta_since`;
  дополнительно считается для сводки/лога (агрегированный INFO, не per-reset —
  сброс рутинен на каждом power-cycle). Undecodable-кадров быть не может: демон
  отбрасывает их **до** `insert_raw_sample`, так что stored-колонки —
  авторитетный декод (raw_frame не переразбираем).

### CLI — `src/main.rs`
- `tm recompute-segments` — роутится до открытия BLE-адаптера (как `stats`/
  `status`). Печатает сводку: кадров, сессий, диапазон дат, сегментов, сбросов.

### Проверка на живой БД (2026-07-05)
- Покрытие безопасно: `raw_samples` UTC-спан (05:40→10:30) **полностью
  перекрывает** сид-сегменты (07:21→10:25) — pre-raw истории нет, wholesale
  replace ничего не теряет. БД забэкаплена перед прогоном.
- **2 грубых сид-сегмента → 21 тонких** (21399 кадров, 9 сессий, 5 сбросов).
  Суммарный зачтённый объём **идентичен** (12218 шагов до и после) — только
  грануляция улучшилась: большая «тренировка #1» (12209 шагов, 2h26m) разбилась
  на реальные под-сегменты с внутренними паузами/сходами (напр. 3468-шаговый
  07:23→07:57, затем разрыв до 08:01 и т.д.).
- `tm stats` при дефолтном gap=15 пере-склеивает 21 сегмент обратно в те же 2
  тренировки (12209 + 9) — отображение не изменилось, но теперь смена
  `workout_gap_minutes` реально разбивает по восстановленной под-структуре
  (раньше на 1 коарс-сегменте это было невозможно). `daily_stats` (12266) не
  тронут. `tm widget` корректно пуст (дорожка отключена).
