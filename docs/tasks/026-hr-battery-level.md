# 026 — HR-датчик: уровень заряда батареи

**Статус:** implemented (код + тесты + доки), ожидает финального прогона на
живом железе после сборки
**Дата:** 2026-07-09
**Зависит от:** 025 (heart-rate support — тот же второй BLE-линк)

## Цель

Показывать заряд батареи нагрудного HR-датчика (Polar H10):
1. Краткая пометка в tmux-виджете **только когда заряд низкий** — не раздувать
   обычный вид.
2. Точный процент по консольной команде (`tm status`) в любой момент.

## Протокол

Стандартный BLE **Battery Service `0x180F`**, характеристика **Battery Level
`0x2A19`** (`uint8`, 0–100%). У Polar-устройств — только **Read** (обязательно),
notify не используется практически (значение не пушится непрерывно) — нужно
периодически перечитывать явно.

## Насколько часто опрашивать

**Важный вывод из матчасти:** сама операция чтения (1 байт по уже открытому
GATT-подключению) стоит батарее H10 исчезающе мало на фоне спецификации
**~400 часов** активной работы от несменной CR2025 (это ~год-полтора при
обычных тренировках). То есть часто/редко опрашивать — **не вопрос экономии
батареи датчика**: даже опрос раз в секунду был бы для неё незаметен.

Причина сделать интервал редким — **не батарея датчика**, а:
- проценты меняются медленно, частый опрос — просто лишняя работа без пользы;
- держим `daemon_status` актуальным без необходимости.

Дефолт, согласованный с оператором: перечитывать **раз в час** (плюс сразу
при подключении датчика), и **раз в 30 минут**, если последний известный
заряд ≤ порога (20%) — не ради батареи, а чтобы предупреждение о разрядке не
протухало на целый час.

## Реализация

- `src/hr.rs` — `BATTERY_SERVICE (0x180F)`, `BATTERY_LEVEL (0x2A19)`.
- `src/scan.rs` — `read_hr_battery(peripheral) -> Option<u8>`, разовое
  bounded-чтение (best-effort, WARN на неудаче/таймауте).
- `src/daemon.rs`:
  - Читает заряд сразу после успешного HR-коннекта (в том же spawned-таске,
    что и `connect_hr`/`subscribe_hr`, — не блокирует основной цикл).
  - Периодическая перепроверка: раз в 5 минут дешёвый чек «не пора ли
    перечитать» (по аналогии с `config_tick`/mtime), сам перечит — раз в
    60 мин (или 30 мин при заряде ≤20%). Bounded read, инлайн в select! —
    не требует отдельной spawned-таски (частота слишком низкая, чтобы
    беспокоиться о блокировке).
  - Сброс `hr_battery_pct`/`hr_battery_last_read` при потере HR-линка —
    протухший процент от снятого датчика не должен висеть в статусе.
- `src/store.rs` — колонка `hr_battery_pct` в `daemon_status` (ALTER,
  `Option<i64>`).
- `tm status` — `heart rate: sensor connected, 63 bpm, battery 42%` (заряд
  опускается, если ещё не прочитан).
- `tm widget` — 10-е TSV-поле `hr_battery_pct`, всегда отдаёт сырой процент
  (пусто — не прочитан); **сам tmux-скрипт** решает показывать иконку
  предупреждения только при ≤20% — контракт данных общий, отображение —
  дело презентации (как и остальной виджет).
- `tm hr` (диагностика) — печатает заряд один раз при подключении, до
  потока bpm.

## Виджет: как выглядит пометка

Обычно (заряд в порядке или неизвестен) — без изменений:

```
  63
```

При низком заряде (≤20%) — маленькая иконка-предупреждение (`nf-md-battery_alert`,
`U+F0083`) в акцентном красном (`#ff5555`, не зависит от фона пилюли) сразу
после bpm:

```
  63 <иконка-батарея-alert>
```

Никакого процента в самом виджете — деталь по запросу через `tm status`.

## Тесты

- `hr.rs`/`scan.rs`: логика чтения покрывается интеграционно на реальном
  железе (нет юнит-тестов для BLE I/O, как и для остального `scan.rs`).
- `daemon.rs`: чистая функция выбора интервала (`hr_battery_poll_interval`,
  60/30 мин по порогу) — юнит-тест без клока/BLE.
- `store.rs`: roundtrip `hr_battery_pct` в `daemon_status` (аналогично
  остальным полям снапшота).
