# 031 — Залипший `connected` после выключения дорожки (виджет висит вечно)

## Симптом

Дорожка физически выключена, телеметрия не идёт, но tmux-виджет продолжает
показывать pill (`paused` + дневные тоталы), и `tm status` говорит
`treadmill: connected`. Состояние не рассасывается само — висит до рестарта
демона.

## Диагноз (по `daemon_status` + `raw_samples`/`hr_samples`)

Снято 2026-07-09 18:43, дорожка выключилась в 18:24:

```
last raw sample (дорожка): 18:24:37   (~19 мин назад)
raw_samples  за 5 мин:     0
hr_samples   за 5 мин:     297        ← ~1/сек
daemon_status.connected =  1
daemon_status.updated_at = 18:43      ← свежий heartbeat
```

Поток `0x2ACD` мёртв 19 минут, а флаг `connected` не сброшен. `tm widget`
печатает pill по контракту (задача 009: «connected + свежий heartbeat»), tmux
честно его рисует. Виджет и tmux-фильтр сессий (задача 008 в `ankor-dotfiles`)
ни при чём — корень в демоне.

От ровно этого сценария защищали **два независимых** механизма. Оба
нейтрализованы одной и той же ошибкой: **«event-loop жив» перепутано с
«телеметрия дорожки идёт»**.

### 1. `NOTIFICATION_TIMEOUT` (20с) никогда не набегает

`src/daemon.rs:671`, внутри `tokio::select!`:

```rust
notification = tokio::time::timeout(NOTIFICATION_TIMEOUT, notifications.next()) => {
```

`select!` конструирует выражение каждой ветки заново на **каждой** итерации
`loop`. Как только любая другая ветка завершается первой, `select!` отдаёт
управление, цикл проворачивается, и `timeout(...)` создаётся с нуля — отсчёт
20с стартует заново, накопленное состояние таймера теряется.

Соседние ветки тикают заметно чаще 20с:

- `command_tick` — `CONTROL_POLL_INTERVAL` = **1с** (`daemon.rs:606`, задача 013);
- HR-стрим — ~1 кадр/с при надетом датчике (задача 025);
- `config_tick` — `CONFIG_RELOAD_INTERVAL` = 5с (`daemon.rs:610`, задача 017).

Значит таймер сбрасывается минимум раз в секунду и до 20с не доживает
**никогда** — ветка `Err(_) => break` (`daemon.rs:676`, «no telemetry received
— treating as disconnected») недостижима, пока крутится control-poll.

Родственник грабли из задачи «tokio::select! precondition unwrap hazard», но
не она: там проблема в вычислении выражения ветки, здесь — в потере состояния
таймера при пересоздании future.

### 2. Watchdog задачи 018 сторожит не то

`STREAMING_STALE_THRESHOLD` = 30с задуман как тайт-порог именно на молчание
дорожки — комментарий у константы прямо говорит «a connected treadmill is never
silent that long» (`daemon.rs:131`). Но `Watchdog::touch()` вызывается из
`State::persist()` (`daemon.rs:1805`), а `persist()` дёргает **любая** ветка
цикла — 24 call site в файле: HR-кадр, control-poll, power-событие, конфиг.

Пульсометр надет и шлёт ~1 bpm/с, каждый кадр персистится → `touch()` раз в
секунду → watchdog вечно считает всё здоровым. Фактически он сторожит живость
event-loop (для чего и создавался в инциденте 2026-07-05 с зависшим
CoreBluetooth), а не поток `0x2ACD`. Тайт-порог в 30с — фикция.

### Почему вылезло только сейчас

Без надетого HR-датчика `command_tick` (1с) всё равно сбрасывал бы
`NOTIFICATION_TIMEOUT`, но watchdog при отсутствии HR-кадров имел шанс
протухнуть и перезапустить процесс, маскируя баг №1. Polar H10 (задача 025)
держит цикл горячим и добивает вторую линию обороны — оба механизма молчат
одновременно.

## План фикса

1. **Абсолютный дедлайн вместо relative-timeout.** Хранить
   `last_sample: Instant`, обновлять его при разборе кадра `0x2ACD`. Ветку
   заменить на `_ = tokio::time::sleep_until(last_sample + NOTIFICATION_TIMEOUT)`
   → `break`. `sleep_until` можно пересоздавать хоть каждую итерацию: дедлайн
   абсолютный, он не «отъезжает». Ветку `notifications.next()` вернуть в
   `select!` отдельным армом.
2. **Развести watchdog.** `touch()` не должен ехать на `persist()`. Ввести
   отдельный `touch_telemetry()`, вызываемый **только** при разборе кадра
   `0x2ACD`, и именно его сверять со `STREAMING_STALE_THRESHOLD`, пока
   `streaming == true`. Общий `touch()`/`WATCHDOG_STALE_THRESHOLD` остаётся
   сторожем живости цикла — это отдельная, по-прежнему нужная гарантия.

Оба пункта независимы и каждый сам по себе закрывает симптом; делать надо оба —
это две разные гарантии (быстрый graceful reconnect vs. self-healing рестарт).

## Проверка

- Юнит-тест на `sleep_until`-семантику не выйдет без часов — гонять через
  `tokio::time::pause()`/`advance()`: убедиться, что при ветке, тикающей раз в
  1с, дедлайн телеметрии всё равно наступает через 20с.
- Тест на watchdog: `touch()` (persist) не должен продлевать
  `stale_threshold()` в streaming-фазе; только `touch_telemetry()`.
- Живая проверка: выключить дорожку с надетым HR-датчиком → в течение ~20с
  `tm status` показывает `not connected`, виджет исчезает, датчик остаётся
  подключённым (HR-линк независим, задача 025).
