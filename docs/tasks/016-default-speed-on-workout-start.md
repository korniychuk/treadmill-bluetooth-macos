# 016 — Расчётная дефолтная скорость на старте тренировки

## Проблема

Дорожка при каждом включении/сбросе выставляет ленту на заводской минимум
(~0.5 км/ч). Оператор ходит на ~2.5 и каждый раз доводит вручную. Задача 012
уже решает это для паузы **внутри** живой сессии (`Paused → Walking` с захваченной
`pre_pause_speed`). Не покрыт **старт новой тренировки**, где восстанавливать
нечего — pre-pause скорости ещё нет.

Нужно: на старте тренировки выставить **расчётную** дефолтную скорость (не из
конфига — вычисляется из истории). Конфиг для «стандартной скорости» не нужен.

## Значение скорости — trimmed mean последней подходящей тренировки

Дефолт = «крейсерская» скорость последней **подходящей** тренировки:

- **Подходящая** = зачтённое время ходьбы (`walking_time_s`, без пауз/away) **≥ 30 мин**.
  (Решение оператора: реальная ходьба, а не wall-clock со стенами-паузами.)
- **Область поиска** — вся история, most-recent-first. Первая тренировка дня
  берёт эталон из вчерашней/последней подходящей; вторая за день — из первой
  (если та ≥ 30 мин). Текущая (только начатая) тренировка имеет ~0 walking_time
  → не проходит порог → не мешает.
- **Trimmed mean**: берём сэмплы скорости ленты этой тренировки (из `raw_samples`),
  фильтруем `>= WALKING_FLOOR (0.8)` (отсекаем рампу/паузы/минимум), сортируем,
  **отрезаем по 15% сверху и снизу** (остаётся средние 70%), берём
  **арифметическое среднее**, округляем до 0.1 км/ч.
  - Если после trim ничего не осталось (очень мало сэмплов) — среднее по всем
    отфильтрованным (fallback).
  - Нет ни одного сэмпла ≥ floor → эта тренировка не даёт значения, идём к более
    старой подходящей (WARN).

Чистая функция `trimmed_mean_speed(speeds, floor, trim) -> Option<TrimmedSpeed>`
(unit-тесты), оркестровка `compute_default_speed(store, gap) -> Option<DefaultSpeed>`.

## Триггер — когда применять

На переходе **в `Walking`, где нет `pre_pause_speed`** для восстановления
(задача 012 владеет случаем «есть что восстановить»). Два входа:

1. **`Unknown → Walking`** — подключились, лента уже едет (**сценарий 1**).
2. **`Paused → Walking` с `pre_pause_speed == None`** — мы видели ленту
   остановленной, потом она стартовала на заводском минимуме (**сценарии 2 и 3**:
   старт с пульта после коннекта; реконнект после розетки/сброса устройства).

`AwayWhileRunning → Walking` не трогаем (оператор вернулся, лента уже на его
скорости; guard ниже всё равно отсечёт).

### Ключевой guard: лента должна быть на crawl (`<= 0.8`)

Применяем **только если текущая скорость ленты `<= DEFAULT_SPEED_APPLY_CEILING (0.8)`**
— т.е. устройство только что стартовало/сбросилось на заводской минимум. Если
лента уже едет на реальной скорости — оператор её задал сам (или это перезапуск
демона посреди ходьбы) → **не трогаем**.

Почему guard, а не «не было тренировки 15 минут»: сценарии оператора
противоречат чисто-временно́му правилу (сценарий 3 хочет применение в пределах
1 минуты после сброса). Настоящий сигнал — «лента на бесполезном заводском
дефолте, потому что только что (пере)стартовала» — детектится напрямую как
`speed <= crawl`. Это же делает фичу безопасной против перезапуска демона
посреди ходьбы (лента на 3.5 > 0.8 → skip, не сбрасываем).

| Ситуация | Скорость ленты | Действие |
|---|---|---|
| Сценарий 1: коннект, лента только включилась | 0.5 | set 2.5 ✅ |
| Сценарий 1: коннект, оператор уже задал 3.0 | 3.0 | skip ✅ |
| Сценарий 2/3: старт/резюм на заводском минимуме | 0.5 | set 2.5 ✅ |
| Перезапуск демона посреди ходьбы | 3.5 | skip ✅ |

### Прочие guard'ы

- **Раз за сессию** (`default_speed_applied: bool`, свежий на каждую BLE-сессию,
  как локалы `stream_with_presence`). Реконнект/новая сессия сбрасывают флаг →
  каждая физическая (пере)тренировка обрабатывается. Одна попытка за сессию
  (ставим флаг при попытке с валидной целью, любой исход) — чтобы не устроить
  retry-шторм при флапе presence на старте.
- **Нет подходящей тренировки** (первый запуск / всё < 30 мин) → лента остаётся
  на заводском дефолте, ничего не пишем (INFO). Конфиг-дефолта нет by design.

### BLE-запись — bounded (конвенция watchdog, задача 007)

Переиспользуем `restore_speed` (take-control + set_speed) и `SPEED_RESTORE_TIMEOUT`
(15s) из задачи 012. Ошибка/timeout write → WARN, демон не падает, лента остаётся
как есть. Это легитимное BLE-управление (разрешено), НЕ прошивочное изменение.

## Toast

`notify::default_speed_applied(from, to)` — одна строка `Set your usual pace
0.5 → 2.5 km/h`. Фичево отделено от «Speed restored» задачи 012 (это дефолт, не
восстановление). В `Paused → Walking`-default показываем именно его (без
pause-длительности — «пауза» в сценарии 3 = сброс устройства, длительность
бессмысленна).

## CLI

`tm default-speed` — печатает расчётный дефолт и из какой тренировки взят
(диагностика; read-only, без BLE). Если подходящей нет — сообщает, что лента
осталась бы на заводском дефолте.

## Замечания

- Это удобство, не замок: наша set-speed — one-shot; оператор в любой момент
  доводит скорость пультом.
- `walking_speeds_in_window` включает away-периоды (лента едет, шагов нет) —
  скорость там всё равно валидная «заданная»; trim гасит выбросы. Реконструкция
  presence по-сэмпльно (как в recompute) избыточна для оценки.
