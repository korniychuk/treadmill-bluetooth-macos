# 014 — конфигурируемый workout-gap, сегменты, две вьюхи статистики

## Мотивация

Порог склейки тренировок (`WORKOUT_GAP_THRESHOLD_S = 15*60`) захардкожен в
`store.rs`. Оператор хочет его конфигурировать. Смена порога «по-хорошему»
требует пересчёта истории — отсюда выбор модели хранения.

## Ключевое решение: хранить порог-независимый грейн (сегменты)

Вместо материализованных `workouts` (порог запечён при записи) храним **сегменты**
и склеиваем в тренировки **на чтении**.

- **Сегмент** = непрерывное **зачтённое** шагание. Граница = presence уходит из
  `Walking` (пауза `speed=0` ИЛИ сход `AwayWhileRunning`). Сегменты дёшевы
  (единицы-десятки в день).
- **Тренировка** (для показа) = склейка соседних сегментов с разрывом ≤
  `workout_gap`. Чистая функция на чтении. Смена порога → мгновенно,
  ретроактивно в обе стороны, без пересчёта, без CPU-нагрузки.
- `raw_samples` **не трогаем** — ground truth, хранится всегда (для будущих
  пересчётов любой логики).
- `daily_stats` **не трогаем** — посэмпловый, календарно-точный,
  порог-независимый (см. вьюхи ниже).

## Полночь / дневная разбивка — строго календарная (без флага)

Ранее обсуждался флаг двух вьюх (по дате начала vs календарно) — **отменён**
оператором. Дневная статистика всегда **строго по календарным суткам (реально за
24h)**: шаги режутся переходом суток. Это **текущее** поведение `daily_stats`, и
оно сохраняется без изменений.

Атрибуция самой **тренировки** (объекта) остаётся по **дате начала**
(`workout.date` = local date первого сегмента) — тоже как сейчас. То есть на
полуночной тренировке дневной итог полуночного дня может не совпадать с суммой
показанных под ним тренировок — это осознанно (итог = реальные шаги суток).

## Цели (goals) — календарные (без изменений)

Реалтайм-вехи «сегодня ≥ 8000 шагов» считают текущий календарный день из
`daily_stats` — согласовано с дневной разбивкой.

## Конфиг

`workout_gap_minutes` в **существующем** `~/.config/treadmill-bluetooth-macos/goals.json`
(рядом с `goals`) — минимум churn, один per-user файл. Дефолт **15**. Валидация:
не число / ≤0 → дефолт + WARN. (Обобщение в `config.json` — возможно позже,
отдельно.)

## Вне scope (отдельные задачи)

- **`away_threshold` (10с) конфигурируемый** — это про **кредит** (что считается
  ходьбой при сходе), не про сегментацию. Ретроактивность = replay из
  `raw_samples`. Отдельная задача.
- Обобщение конфига в `config.json`.

## План реализации

1. **Config**: `goals.rs` (или новый `config.rs`) читает `workout_gap_minutes`;
   пробросить в daemon/stats. Чистый парс + тест.
2. **Схема**: таблица `activity_segments(id, started_at, ended_at, date,
   distance_m, steps, walking_time_s)`; `date` = local date начала сегмента.
   Индекс по дате.
3. **Запись**: демон в presence-переходе закрывает текущий сегмент при уходе из
   `Walking`; `credit_activity` продлевает открытый сегмент. `daily_stats` —
   как есть (посэмплово).
4. **Чтение**: чистая `merge_segments(segments, gap) -> Vec<Workout>` (юнит-тест:
   границы, обе стороны порога, полночь). `workouts_for`/`latest_workout`/`stats`
   переводятся на неё. `daily_stats` и `tm stats` дневные итоги — **как есть**
   (строго календарные; никакого флага вьюх).
5. **Миграция**: существующие `workouts` → сегменты (как одиночные сегменты;
   старое до внедрения не пере-разбивается тоньше, но пере-склеивается порогом —
   истории мало, приемлемо). Старую `workouts` можно оставить как есть или
   дропнуть после сидинга — решить в реализации, сохранно.
6. Тесты, `clippy`, обновить `CLAUDE.md` + этот док.

## Реализация (что сделано)

### Схема
- Новая таблица `activity_segments(id, started_at, ended_at, date, distance_m,
  steps, walking_time_s)` + индекс `idx_activity_segments_date`. `date` = local
  дата **начала** сегмента (атрибуция тренировки по дате старта).
- `daily_stats`, `raw_samples`, `workouts` — **не тронуты**.

### Open/close сегмента (без флага в БД)
- «Открытость» сегмента живёт **в памяти демона** — `current_segment:
  Option<i64>` в `stream_with_presence` (рядом с `pending`/`away_since`). Нет
  колонки `open` — схема ровно как в спеке.
- `credit_activity(steps, dist, time, now, open_segment) -> segment_id`:
  `Some(id)` продлевает сегмент, `None` открывает новый; возвращает id, демон
  кладёт его обратно в `current_segment`. daily_stats + сегмент в **одной
  транзакции** (как раньше workouts). Stale-id (сегмент не найден, `rows==0`)
  → WARN + новый сегмент (кредит не теряется).
- **Close** — в presence-переходе `daemon.rs`: `if next_state != Walking {
  current_segment = None; }`. Никакой записи в БД — следующий шаг откроет новый
  сегмент.
- **Restart-safety**: `current_segment` свежий на каждую сессию (как
  `device_baseline`/session-логика). Рестарт посреди ходьбы просто открывает
  новый сегмент; read-time `merge_segments` пере-склеивает его с досегментом,
  если разрыв ≤ порога. Split при рестарте самозалечивается на чтении.

### Чтение
- Чистая `merge_segments(&[Segment], gap_minutes) -> Vec<Workout>` — склейка
  соседних сегментов при `next.started_at − prev.ended_at ≤ gap` (граница
  **включительная** `<=`, как старый split, чтобы сид пере-склеивался 1-в-1 при
  gap=15). `id` тренировки = id её первого сегмента (стабилен, для `#N` и
  in-progress маркера). Юнит-тесты: пусто, склейка/сплит, ровно-на-границе,
  ретроактивный regroup (gap 5 vs 15 vs 30), полночь.
- `workouts_for(date, gap)` / `latest_workout(gap)` грузят **все** сегменты
  (их мало) и мёржат в памяти, потом фильтруют по дате старта. Кросс-полночь:
  мёрж идёт по всей истории до фильтра, чтобы до-полуночная тренировка вобрала
  после-полуночные сегменты.

### Конфиг
- `workout_gap_minutes` в `~/.config/treadmill-bluetooth-macos/goals.json`
  (рядом с `goals`). Дефолт 15. Чистый парс `read_workout_gap_minutes` →
  `GapSetting { Configured(n) | Invalid | Unset }` (юнит-тест). Загружается
  **на чтении** (`stats`/`status`/`widget`), демону при записи не нужен.
- **Логирование (решение по open-question):** ключ **отсутствует** →
  тихо дефолт (это норма: каждый до-014 конфиг без ключа, а `widget` читает
  раз в ~2 с — WARN был бы happy-path-спамом, запрещённым конвенцией);
  файла нет → тихо дефолт; ключ **есть, но не положительное целое** → WARN +
  дефолт. Это уточняет спековское «Invalid/missing → WARN»: под «missing»
  понимаем невалидное значение, не отсутствие ключа. Формально — согласовать на
  ревью, но выбор продиктован правилом «логируем аномалии, не happy path».

### Widget (доп. scope от координатора, задача 014)
- TSV расширен до **8 полей**: `state, workout_count, cur_walking_s, cur_steps,
  cur_distance_m, day_walking_s, day_steps, day_distance_m`. `workout_count` —
  число сегодняшних merged-тренировок (отражает `workout_gap_minutes`);
  `cur_*` — текущая тренировка; `day_*` — `daily_stats`. Контракт — см.
  `docs/tasks/009`. Shell-парсер в отдельном dotfiles-репо (не здесь).

### Миграция и судьба `workouts`
- Сид `activity_segments` из `workouts` был **временной** миграцией 014 (каждая
  старая тренировка → один грубый сегмент). Его **заменил** `tm recompute-segments`
  (задача 015): пересборка из `raw_samples` в полном грейне, строго лучше.
- **`workouts` УДАЛЕНА** (таблица дропнута, CREATE + seed-миграция убраны из
  `store.rs`) — после ревью оператором и recompute из raw. Путь апгрейда для
  будущих существующих юзеров теперь = `tm recompute-segments` (или просто свежие
  сегменты от демона). Бэкапы БД перед дропом оставлены для отката.

## Решено (ранее открытые)

- **`workouts`: дропнута** (не архив) — см. выше.
- Логирование absent-ключа `workout_gap_minutes` как тихого дефолта — принято
  (правило «логируем аномалии, не happy path»; `widget` поллит ~2с).
- Дефолт порога **15 мин**.
