# 014 — конфигурируемый workout-gap, сегменты, две вьюхи статистики

## Мотивация

Порог склейки тренировок (`WORKOUT_GAP_THRESHOLD_S = 15*60`) захардкожен в
`store.rs`. Оператор хочет его конфигурировать. Смена порога «по-хорошему»
требует пересчёта истории — отсюда выбор модели хранения.

## Ключевое решение: хранить порог-независимый грейн (сегменты)

Вместо материализованных `workouts` (порог запечён при записи) храним **сегменты**
и склеиваем в тренировки **на чтении**.

- **Сегмент** = непрерывное **зачтённое** шагание. Граница = presence уходит из
  `Walking` (пауза `speed=0` ИЛИ сход `AwayWhileRunning`). Сегменты дёшевы
  (единицы-десятки в день).
- **Тренировка** (для показа) = склейка соседних сегментов с разрывом ≤
  `workout_gap`. Чистая функция на чтении. Смена порога → мгновенно,
  ретроактивно в обе стороны, без пересчёта, без CPU-нагрузки.
- `raw_samples` **не трогаем** — ground truth, хранится всегда (для будущих
  пересчётов любой логики).
- `daily_stats` **не трогаем** — посэмпловый, календарно-точный,
  порог-независимый (см. вьюхи ниже).

## Полночь / дневная разбивка — строго календарная (без флага)

Ранее обсуждался флаг двух вьюх (по дате начала vs календарно) — **отменён**
оператором. Дневная статистика всегда **строго по календарным суткам (реально за
24h)**: шаги режутся переходом суток. Это **текущее** поведение `daily_stats`, и
оно сохраняется без изменений.

Атрибуция самой **тренировки** (объекта) остаётся по **дате начала**
(`workout.date` = local date первого сегмента) — тоже как сейчас. То есть на
полуночной тренировке дневной итог полуночного дня может не совпадать с суммой
показанных под ним тренировок — это осознанно (итог = реальные шаги суток).

## Цели (goals) — календарные (без изменений)

Реалтайм-вехи «сегодня ≥ 8000 шагов» считают текущий календарный день из
`daily_stats` — согласовано с дневной разбивкой.

## Конфиг

`workout_gap_minutes` в **существующем** `~/.config/treadmill-bluetooth-macos/goals.json`
(рядом с `goals`) — минимум churn, один per-user файл. Дефолт **15**. Валидация:
не число / ≤0 → дефолт + WARN. (Обобщение в `config.json` — возможно позже,
отдельно.)

## Вне scope (отдельные задачи)

- **`away_threshold` (10с) конфигурируемый** — это про **кредит** (что считается
  ходьбой при сходе), не про сегментацию. Ретроактивность = replay из
  `raw_samples`. Отдельная задача.
- Обобщение конфига в `config.json`.

## План реализации

1. **Config**: `goals.rs` (или новый `config.rs`) читает `workout_gap_minutes`;
   пробросить в daemon/stats. Чистый парс + тест.
2. **Схема**: таблица `activity_segments(id, started_at, ended_at, date,
   distance_m, steps, walking_time_s)`; `date` = local date начала сегмента.
   Индекс по дате.
3. **Запись**: демон в presence-переходе закрывает текущий сегмент при уходе из
   `Walking`; `credit_activity` продлевает открытый сегмент. `daily_stats` —
   как есть (посэмплово).
4. **Чтение**: чистая `merge_segments(segments, gap) -> Vec<Workout>` (юнит-тест:
   границы, обе стороны порога, полночь). `workouts_for`/`latest_workout`/`stats`
   переводятся на неё. `daily_stats` и `tm stats` дневные итоги — **как есть**
   (строго календарные; никакого флага вьюх).
5. **Миграция**: существующие `workouts` → сегменты (как одиночные сегменты;
   старое до внедрения не пере-разбивается тоньше, но пере-склеивается порогом —
   истории мало, приемлемо). Старую `workouts` можно оставить как есть или
   дропнуть после сидинга — решить в реализации, сохранно.
6. Тесты, `clippy`, обновить `CLAUDE.md` + этот док.
